背景

##利用huskey和lint-staged在git commit之前做代码检查

git工作流中提供了一系列钩子给用户做自定义操作，分布在.git/hooks文件夹里面，默认都是未启用状态。

为了保证提交到远程分支的代码的可靠性，可以在commit-msg hook里面对提交的代码做静态分析，不通过则无法提交。

  1. 安装husky和lint-staged
  1. 配置package.json

husky在install action中会替换hooks目录下的相关钩子函数，触发时转去执行package.json中配置的命名，在uninstall action中会恢复原状。

lint-staged是为了提高lint的效率，只知道提交的文件进行检查。

##前端代码查重–jsinspect

重构代码时很重要的一步就是抽取通用代码，不要repeat yourself，不要ctrl-c/ctrl-v。

jsinspect帮助我们快速查找项目里面的重复代码。

参数解析：

  • Threshold indicates the minimum number of nodes to analyze.

    • Identifiers indicates whether or not the nodes in a match should also havematching identifiers
    • literals means whether or not literal values should match
    • minInstances specifies the min number of instances for a match

jsinspect的基本思路如下图所示：

首先通过对每个文件通过babel parser生成AST，对AST进行深度遍历保存下nodes全序列。根据设定的threashold值遍历所有满足条件的node子序列，它们就是要匹配的潜在对象，每一个子序列生成一个key，计算方法就是n个node的type和length相连接。对相同key值的子序列聚合得到相应的matches。

jsinspect官方源的babel parser是6.x版本的，所以只支持js，jsx和flow的translate，而我们项目里有很多ts，也需要支持。babel 7开始支持typescript，所以需要对jsinspect稍作修改。

  1. package.json

    “babylon”: “6.16.1” => “@babel/parser”: “^7.3.4”,

  2. bin/jsinspect

    var extensions = [‘.js’, ‘.jsx’, ‘.ts’]; //增加ts支持

  3. lib/parser

    将plugins中的flow替换成typescript，两者不能同时支持

详细的见https://github.com/JoeMaybe/jsinspect/tree/feature_ts_support

Posted on 2019-02-19

认识Javascript的类

Javascript中的类本质上是一个构造函数（User）与一个用于在该类（User.prototype）实例间共享方法的原型对象的结合

在JS中，函数本身也是一个对象，prototype是它的一个属性，称为对象原型，其本质也是一个对象，包含constructor和其他属性成员。constructor默认指向自身构造函数。 当我们使用new操作符创建User的实例时，产生的对象u得到了自动分配的原型对象，该原型对象被存储在User.prototype中。实例对象u根据prototype.constructor 所指向的构造函数进行实例初始化。

认识Javascript的继承
